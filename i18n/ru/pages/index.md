  <h2>Что такое FxDart?</h2>
  <p>
    FxDart переносит модель программирования FxTS в Dart: набор из ~120 небольших
    композируемых функций для работы с коллекциями и асинхронными данными.
    В его основе три идеи:
  </p>
  <ul>
    <li><strong>Ленивые вычисления</strong> — операторы вроде <code>map</code>,
      <code>filter</code> и <code>take</code> лишь строят конвейер и не делают работы,
      пока терминальный оператор (<code>toList</code>, <code>each</code>,
      <code>reduce</code>…) не протянет через него значения. Обработка диапазона
      из миллиона элементов с <code>.take(3)</code> вычислит ровно 3 результата.</li>
    <li><strong>Единая модель для sync и async</strong> — одни и те же имена операторов
      работают и с обычными <code>Iterable</code>, и с <code>FxAsyncIterable</code>,
      pull-основанной асинхронной последовательностью FxDart (с мостами к <code>Stream</code>
      в обе стороны).</li>
    <li><strong>Декларативная конкурентность</strong> — <code>concurrent(n)</code>
      просит <em>вышестоящий</em> конвейер вычислять по <code>n</code> элементов
      за раз, сохраняя порядок результатов. Это фирменная возможность FxTS,
      точно перенесённая в Dart.</li>
  </ul>

  <h2>Зачем это нужно?</h2>
  <p>
    В Dart уже есть <code>Iterable</code> и <code>Stream</code>. FxDart оправдывает
    себя там, где их возможностей не хватает:
  </p>
  <table>
    <tr><th>Задача</th><th>Чистый Dart</th><th>FxDart</th></tr>
    <tr>
      <td>Ограничить число одновременных вызовов API до <em>n</em>, сохранив порядок</td>
      <td>Ручные очереди, <code>Completer</code>, кропотливый учёт</td>
      <td><code>.map(fetch).concurrent(3)</code></td>
    </tr>
    <tr>
      <td>Асинхронные конвейеры преобразований</td>
      <td><code>Stream</code> работает по push-модели; смешивать <code>await</code>, backpressure и ленивость становится непросто</td>
      <td>Pull-цепочка — каждое значение вычисляется только по запросу</td>
    </tr>
    <tr>
      <td>Обработка данных (группировка, индексация, подсчёт, разбиение, zip, chunk…)</td>
      <td>Каждый раз самописные циклы</td>
      <td>По одной проверенной именованной функции на каждое понятие</td>
    </tr>
    <tr>
      <td>Читаемые многошаговые преобразования</td>
      <td>Вложенные вызовы или промежуточные переменные</td>
      <td>Цепочки <code>fx()</code> слева направо, полностью типизированные</td>
    </tr>
  </table>

  <h2>Плюсы &amp; минусы</h2>
  <div class="proscons">
    <div class="box pros">
      <h3>✓ Плюсы</h3>
      <ul>
        <li><strong>Ленивость бесплатно</strong> — конвейеры прерываются досрочно; вычисляются только запрошенные значения.</li>
        <li><strong>Конкурентность с сохранением порядка</strong> одним оператором: <code>concurrent(n)</code> / <code>concurrentPool(n)</code> в порядке завершения.</li>
        <li><strong>Полностью типизированные цепочки</strong> — <code>fx()</code> сохраняет вывод типов от начала до конца; синхронные операторы — обычные функции над нативными <code>Iterable</code>, поэтому всё совместимо с обычным Dart.</li>
        <li><strong>Небольшие узкоспециализированные функции</strong> — ~120 операторов: transform / filter / slice / combine / aggregate / object / util.</li>
        <li><strong>Проверенная в бою семантика</strong> — поведение перенесено из FxTS вместе с 850+ его тестов.</li>
        <li><strong>Ноль зависимостей</strong> — чистый Dart.</li>
      </ul>
    </div>
    <div class="box cons">
      <h3>✗ Минусы</h3>
      <ul>
        <li><strong>Нет каррирования с данными в конце</strong> — в Dart нет перегрузки функций, поэтому каррированный стиль <code>pipe</code> из FxTS превращается в цепочки; динамический <code>pipe()</code> теряет статические типы.</li>
        <li><strong>Вторая асинхронная абстракция</strong> — <code>FxAsyncIterable</code> существует потому, что <code>Stream</code> не умеет выражать обратный канал для конкурентности; мосты просты, но это ещё одно понятие для изучения.</li>
        <li><strong>Порог входа</strong> — мышление ленивыми конвейерами отличается от императивных циклов.</li>
        <li><strong>Не всегда самый быстрый вариант</strong> — на маленьких горячих циклах написанный вручную <code>for</code> может обогнать композицию операторов; FxDart оптимизирован под ясность и работу с I/O.</li>
        <li><strong>Некоторые API из TS не переносятся буквально</strong> — вместо этого они получают нативные для Dart формы: <code>curry</code> стал типизированным геттером-расширением <a href="tutorials/curried.html"><code>.curried</code></a>, а старые имена оставлены как deprecated-заглушки для миграции.</li>
      </ul>
    </div>
  </div>
