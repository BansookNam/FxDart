flatMap

Maps each element to an iterable, then flattens the results one level — the same contract as Iterable.expand.

Iterable<B> flatMap<A, B>(Iterable<B> Function(A a) f, Iterable<A> iterable) FxAsyncIterable<B> flatMapAsync<A, B>(FutureOr<Iterable<B>> Function(A a) f, FxAsyncIterable<A> iterable) Fx<R> Fx.flatMap<R>(Iterable<R> Function(T a) f) // chain FxAsync<R> FxAsync.flatMap<R>(FutureOr<Iterable<R>> Function(T a) f)

Lecture

flatMap is map followed by one level of flattening: each element is turned into a collection of results, and those collections are spliced together into a single flat lazy sequence. It's the tool for "one input, many outputs" — splitting a sentence into words, expanding a user into their orders, turning a range into pairs.

FxTS deviation: in FxTS, the callback can return any mix of plain values and iterables and flatMap figures out what to flatten via DeepFlat type magic. Dart has no equivalent of that conditional type, so the Dart port requires f to always return an Iterable<B> — exactly like Iterable.expand. Return a single-element list ([x]) to emit exactly one value per input, or an empty list to emit none.

On the async side, flatMapAsync's internal state machine has to track "which sub-iterable am I currently draining" between pulls, so it consumes its upstream serially — wrapping it in .concurrent(n) only speeds up pulling already-available items, not an await that happens inside the callback itself. If you need concurrent async work per element, do that work in a .map(...).concurrent(n) stage first, then .flatMap((list) => list) to flatten the already-resolved lists — see Demo 2.
